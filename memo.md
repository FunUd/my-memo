# LLMコンテキスト及びトークンコスト最適化 総合ガイド
LLM（大規模言語モデル）のコンテキストウィンドウが拡大する一方で、プロダクション環境や自律型AIエージェントの運用、大規模な開発アシスタントシステムにおける「トークンコスト」と「レイテンシ」の増大は極めて深刻な課題です。
本ドキュメントでは、各先進事例やオープンソースツール、実務的な開発知見に基づき、**「コンテキストキャッシュの仕組み」「トークンコスト節約ノウハウ」「トークンコスト節約ツール」**の3つの観点から最適化戦略を網羅的に解説します。
## 1. コンテキストキャッシュの仕組み
コンテキストキャッシュ（Context Caching / Prompt Caching）は、LLMプロバイダ側で共通・重複するプロンプト入力をキャッシュし、2回目以降の入力処理（Inputプレフィックスの計算）のコストとレイテンシを大幅に削減する技術です。
### 1.1 基本原理とコストメリット
 * **KVキャッシュの再利用**: LLMの推論において、過去のトークンから生成された「Key-Value（KV）ベクトル」をサーバー側に保持し、次回リクエスト時に再利用します。
 * **料金構造の優位性**:
   * プロバイダ（AnthropicのClaudeやOpenAIなど）によって異なりますが、キャッシュにヒットしたトークン（Cache Hits）の入力単価は、通常の入力単価（Cache Misses）の**約10%〜50%**にまで抑えられます。
   * キャッシュの書き込み（Cache Creation/Write）時には、通常料金に若干のプレミアムが上乗せされる場合がありますが、複数回ヒットさせることでトータルコストは劇的に低下します。
### 1.2 ヒットの条件と構造的制約
 * **完全一致のプレフィックス（前方一致）**:
   * キャッシュは、メッセージシーケンスの**先頭（Prefix）から1文字（1トークン）も違わずに完全に一致している部分**のみに適用されます。
   * 途中に動的なデータ（ユーザーの最新の発言、セッション固有のUUID、現在時刻、ランダムなシード値など）が1箇所でも混入すると、**それ以降のコンテキストはすべてキャッシュ対象外（Cache Miss）**になります。
 * **ブロック単位の生存期間（TTL）**:
   * プロバイダ側でキャッシュが保持される時間には制限（数分〜数十分）があり、一定時間リクエストがない場合は破棄されます。また、一般的に最小キャッシュ対象トークン数（例: 1,024トークン以上など）の閾値が設定されています。
## 2. トークンコスト節約ノウハウ
コンテキストキャッシュの特性を理解した上で、アプリケーションのコードやプロンプトの組み立て方を「キャッシュフレンドリー」に設計するための実践的ノウハウです。
### 2.1 決定論的なプロンプト配置（静的・動的データの分離）
キャッシュの「前方一致」の制約をクリアするため、メッセージの配列構造を厳密に管理します。
 1. **静的データの最前線配置**:
   変更されることのないシステムプロンプト（System Prompt）、固定のタスク指示、大規模なAPI仕様、固定のドキュメント（規約など）は必ず配列の先頭に配置します。
 2. **動的データの後方隔離**:
   ユーザーの入力（User Query）、チャットの履歴、ツール実行の結果など、毎回変わる要素は必ず配列の最後尾に結合します。
 3. **メタデータの排除**:
   システムプロンプト内に Current Time: 2026-06-17 08:11:00 や Session ID: xxxx-xxxx のような動的メタデータを無造作に埋め込むと、システムプロンプト全体のキャッシュが毎リクエストごとに破壊されます。これらはシステムプロンプトから切り離し、ユーザーメッセージの直前か、専用の動的コンテキスト領域に移譲する必要があります。
## 3. トークンコスト節約ツール
近年、上記のような「キャッシュのアライメント調整」や「コンテキストの可逆圧縮」を自動化・抽象化するための強力なオープンソースツール（OSS）やライブラリが登場しています。
### 3.1 Headroom
GitHubで大きな注目を集めている、LLMアプリケーションとプロバイダの間に位置する**「コンテキスト最適化レイヤー（プロキシ / ライブラリ）」**です。
 * **コア機能**:
   * AIエージェントが読み込む大量のツール出力、ビルドログ、大規模なJSON配列、リポジトリの検索結果などを、LLMに送信される直前に**60%〜95%自動圧縮**します。
   * **可逆圧縮（Reversible Compression）**: 意味の骨組み（構造）を維持したままノイズを削ぎ落とし、圧縮されたテキスト内に [1000 items compressed. Retrieve more: hash=abc123] のような復元用マーカーを埋め込みます。LLMがより詳細なデータを必要とした場合、自動的にインジェクトされた headroom_retrieve ツールを呼び出して、ローカルにハッシュ保存された元データをオンデマンドで復元・取得します。
 * **導入手法**:
   * **SDK/インライン利用**: Python/TypeScript対応。既存の messages 配列を compress(messages) に通すだけでトークンを削減。
   * **プロキシサーバー**: ローカルでプロキシを立ち上げ、OpenAI/Anthropic SDKの baseURL を差し替えるだけで、既存コードに一切手を加えずに適用可能。Model Context Protocol（MCP）サーバーとしても動作します。
### 3.2 rtk (Token Killer)
「Token Killer」として開発されている、リクエストトークン（rtk）を効率的に削減・制御するためのユーティリティツールです。
 * **コア機能**:
   * 主にプロンプトバッファの溢れや、エージェントが過去の文脈を引きずりすぎてトークンが肥大化する問題を解決します。
   * 過去のコンテキストから優先度の低い情報（冗長なログなど）を自動的にパージ（間引き）したり、重要な情報のみを構造化して要約文に置換することで、ウィンドウの破綻を防ぎます。
### 3.3 caveman
LLMへのコンテキスト注入時における、**「泥臭いテキスト整形・トークン削減」を肩代わりする軽量な最適化ユーティリティ**です。
 * **コア機能**:
   * 開発者が生データ（ログ、マークダウン、コード、CSV）をLLMのコンテキストにそのまま放り込む前に、トークン効率の良い表現形式へと構造的に事前パニング（平坦化・トークン単位のトリミング）を行います。
   * 意味論的な損失を最小限に抑えつつ、トークン消費の激しいスペース、特殊文字、冗長なネスト構造を機械的にプレプロセスします。
## 4. まとめ：エンジニアリングにおける選択基準
トークンコストの削減は、以下のステップでアプローチを組み合わせることが最も効果的です。
```
[ステップ 1: プロンプト構造の最適化]
   └── 静的データを先頭へ、動的（時間・クエリ）を最後尾へ徹底配置（ノウハウの適用）
        ↓
[ステップ 2: プロバイダのネイティブキャッシュの有効化]
   └── プレフィックスの完全一致を維持し、Cache Hits（料金10-50%）を最大化
        ↓
[ステップ 3: 最適化ツールのレイヤー導入]
   └── エージェント等の大量のツール出力に対し、
       「Headroom」等のOSSを導入してコンテキスト自体を60-95%可逆圧縮
